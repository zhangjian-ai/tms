# TMS Backend 开发规范文档

## 1. 项目架构

### 1.1 技术栈
- **框架**: Spring Boot 2.6.13
- **Java 版本**: JDK 17 (编译目标 1.8)
- **ORM**: MyBatis-Plus 3.5.4
- **数据库**: MySQL + Druid 连接池
- **API 文档**: Knife4j (Swagger) 4.1.0
- **缓存**: Redis
- **文件存储**: MinIO
- **数据库版本管理**: Flyway
- **实时通信**: WebSocket
- **工具库**: Hutool 5.7.3, Fastjson 1.2.80

## 2. 分层架构规范

### 2.1 标准三层架构

**Controller 层** → **Service 层** → **Mapper 层**

#### Controller 层职责
- 接收 HTTP 请求，参数校验
- 调用 Service 层处理业务逻辑
- 返回统一格式的响应（Result）
- 使用 Swagger 注解生成 API 文档

#### Service 层职责
- 实现具体业务逻辑
- 事务管理（@Transactional）
- 调用 Mapper 层进行数据操作
- 业务异常处理和日志记录

#### Mapper 层职责
- 继承 MyBatis-Plus BaseMapper
- 定义复杂 SQL 查询方法
- 数据库 CRUD 操作

### 2.2 模块组织规范

每个业务模块按以下结构组织：
```
module/
├── controller/        # 控制器
├── service/          # 服务接口
│   └── impl/        # 服务实现
├── mapper/          # 数据访问层
└── entities/        # 实体类
    ├── *PO.java    # 持久化对象（数据库表映射）
    ├── *DTO.java   # 数据传输对象（请求参数）
    └── *VO.java    # 视图对象（响应数据）
```

## 3. 命名规范

### 3.1 包命名
- 全部小写，使用点分隔
- 业务模块：`com.seeker.tms.biz.{module}`
- 公共组件：`com.seeker.tms.common.{component}`

### 3.2 类命名
- **PO (Persistent Object)**: `{Entity}PO.java`
  - 示例：`TestCasePO.java`, `DevicePO.java`
  - 使用 `@TableName` 注解指定表名
  
- **DTO (Data Transfer Object)**: `{Entity}{Action}DTO.java`
  - 示例：`CaseAddDTO.java`, `DeviceQueryDTO.java`
  
- **VO (View Object)**: `{Entity}VO.java`
  - 示例：`DeviceVO.java`, `ApiInvokeVO.java`
  
- **Controller**: `{Module}Controller.java`
  - 示例：`TestCaseController.java`
  
- **Service**: `{Module}Service.java` (接口) + `{Module}ServiceImpl.java` (实现)
  - 示例：`TestCaseService.java`, `TestCaseServiceImpl.java`
  
- **Mapper**: `{Entity}Mapper.java`
  - 示例：`TestCaseMapper.java`

### 3.3 方法命名
- Controller 方法：动词开头，描述操作
  - `addTestCase()`, `updateDevice()`, `queryList()`
  
- Service 方法：业务语义化命名
  - `addTestCase()`, `deleteById()`, `getDeviceList()`

### 3.4 变量命名
- 驼峰命名法
- 布尔类型：`is/has/can` 开头
- 集合类型：复数形式或 `list/map` 后缀

## 4. 代码规范

### 4.1 注解使用

#### 类级别注解
```java
@Slf4j                    // 日志
@Service                  // 服务层
@RestController          // 控制器
@AllArgsConstructor      // 构造器注入
@RequestMapping("/path") // 路由前缀
@Api(tags = "模块名称")   // Swagger 文档
```

#### 实体类注解
```java
@Data                    // Getter/Setter
@NoArgsConstructor      // 无参构造
@AllArgsConstructor     // 全参构造
@TableName("table_name") // 表名映射
@TableId(type = IdType.AUTO) // 主键自增
```

#### 方法级别注解
```java
@ApiOperation("操作说明")           // Swagger 文档
@PostMapping("/add")               // POST 请求
@GetMapping("/list")               // GET 请求
@Transactional(rollbackFor = Exception.class) // 事务
```

### 4.2 统一响应格式

使用 `Result<T>` 封装所有接口响应：

```java
// 成功响应（无数据）
return Result.success();

// 成功响应（有数据）
return Result.success(data);

// 失败响应
return Result.fail();

// 自定义响应
return Result.builder(code, msg, data);
```

Result 结构：
```json
{
  "code": 200,
  "msg": "success",
  "data": {}
}
```

### 4.3 异常处理

- 使用 `@ControllerAdvice` 全局异常处理器
- Service 层抛出业务异常
- 记录错误日志：`log.error("错误信息", exception)`

### 4.4 事务管理

```java
@Override
@Transactional(rollbackFor = Exception.class)
public Boolean addTestCase(CaseAddDTO caseAddDTO) {
    // 业务逻辑
}
```

- 所有涉及多表操作的方法必须加事务
- 指定 `rollbackFor = Exception.class` 确保所有异常回滚

### 4.5 日志规范

```java
// 信息日志
log.info("操作成功: {}", param);

// 错误日志
log.error("操作失败: " + message);
log.error("异常信息", exception);

// 调试日志
log.debug("调试信息: {}", data);
```

### 4.6 依赖注入

使用构造器注入（推荐）：
```java
@AllArgsConstructor
@Service
public class TestCaseServiceImpl {
    private final TestCaseMapper testCaseMapper;
}
```

### 4.7 时间处理

统一使用 `LocalDateTime`：
```java
testCasePo.setCreateTime(LocalDateTime.now());
testCasePo.setUpdateTime(LocalDateTime.now());
```

## 10. 开发流程

### 10.1 新增业务模块
1. 创建模块包结构
2. 定义实体类（PO/DTO/VO）
3. 创建 Mapper 接口
4. 实现 Service 接口和实现类
5. 编写 Controller
6. 添加 Swagger 注解
7. 编写单元测试

### 10.2 数据库变更
1. 编写 Flyway 迁移脚本
2. 更新对应的 PO 类
3. 测试迁移脚本

### 10.3 代码提交
1. 遵循 Git 提交规范
2. 提交信息格式：`[模块] 操作: 描述`
3. 示例：`[testcase] feat: 新增测试用例管理功能`

## 11. 测试规范

### 11.1 单元测试
- 位置：`src/test/java`
- 命名：`{Class}Test.java`
- 使用 JUnit 5 + Mockito

### 11.2 测试覆盖
- Service 层核心方法必须有单元测试
- Controller 层建议编写集成测试

## 12. 性能优化建议

### 12.1 数据库优化
- 使用批量操作代替循环插入
- 合理使用索引
- 避免 N+1 查询问题

### 12.2 缓存使用
- 热点数据使用 Redis 缓存
- 设置合理的过期时间

### 12.3 异步处理
- 耗时操作使用异步处理
- 使用线程池管理线程

## 13. 安全规范

### 13.1 参数校验
- 使用 `@Valid` 注解校验请求参数
- DTO 中使用 `@NotNull`, `@NotBlank` 等注解

### 13.2 SQL 注入防护
- 使用 MyBatis-Plus 参数化查询
- 避免拼接 SQL 字符串

### 13.3 敏感信息
- 密码使用加密存储
- 配置文件中的敏感信息使用环境变量

## 14. 常见问题

### 14.1 事务不生效
- 检查方法是否为 public
- 检查是否在同一类内部调用
- 确认已添加 `@Transactional` 注解

### 14.2 MyBatis-Plus 查询为空
- 检查实体类 `@TableName` 注解
- 检查字段名与数据库列名映射

### 14.3 跨域问题
- 已配置 `CorsConfig`，允许所有来源访问
- 生产环境需限制允许的域名
